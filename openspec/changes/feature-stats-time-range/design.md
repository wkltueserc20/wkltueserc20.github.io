## Context

目前統計資料是在 `App.tsx` 中透過 `useMemo` 遍歷最近 7 天產生的。為了支援動態區間，我們需要將「天數」抽離為狀態，並優化計算效率以應對長達 28 天的數據處理。

## Goals / Non-Goals

**Goals:**
- 在統計頁面提供 `7D / 14D / 28D` 的切換選項。
- 確保切換時，所有圖表（奶量、睡眠）同步更新。
- 在 28 天模式下保持 X 軸標籤的易讀性。

**Non-Goals:**
- 不支援自定義起始與結束日期的選取。
- 不改變成長曲線（Growth Chart）的顯示邏輯（其應顯示所有歷史紀錄以觀察完整軌跡）。

## Decisions

### 1. 狀態管理與持久化
- **決策**：在 `App.tsx` 使用 `useState<number>(7)`。
- **理由**：區間選取不需要跨 Session 記憶，保持在記憶體中即可，操作簡單直覺。

### 2. 資料生成邏輯重構
- **策略**：
    ```typescript
    const generateChartData = (days: number, type: RecordType) => {
      // 根據 days 動態迴圈
      for (let i = days - 1; i >= 0; i--) { ... }
    }
    ```
- **優化**：在迴圈前，先對 `records` 進行一次性過濾與排序，減少迴圈內部的計算量。

### 3. UI 組件：Segmented Control
- **樣式**：採用圓角容器內含按鈕的設計。
- **佈局**：放置於 `currentTab === 'stats'` 的最上方，作為全域控制器。
- **CSS**：使用 `flex-1` 平分寬度，並以背景色區分當前選中項。

### 4. 圖表響應式適配
- **決策**：在 28 天模式下，`tick` 字體大小從 `10px` 降至 `8px`。
- **理由**：避免日期標籤互相重疊。

## Risks / Trade-offs

- **[Risk] 效能壓力**：當紀錄筆數非常多（如數千筆）時，每次切換區間觸發的 `reduce` 可能造成輕微卡頓。
    - 🛡️ **Mitigation**: 由於我們是本地 IndexedDB 操作且每日紀錄量不大，28 天數據量約為 200-300 筆紀錄，現代瀏覽器可在毫秒內完成運算。
