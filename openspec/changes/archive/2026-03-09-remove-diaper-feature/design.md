## Context

目前系統在 `App.tsx` 中硬編碼了四種紀錄類型，且 UI 佈局固定為三欄式的統計卡片。移除一項功能需要調整狀態機、條件渲染與數據彙整邏輯。

## Goals / Non-Goals

**Goals:**
- 從型別定義中移除 `diaper`。
- 修改 `stats` 計算邏輯，移除尿/便計數。
- 實現「方案 B」的兩列式統計卡片佈局，並新增「最新成長數據」摘要。
- 確保系統加載舊數據或匯入 CSV 時，能過濾掉 `diaper` 紀錄以防報錯。

**Non-Goals:**
- 不會從 CSV 標頭中移除 `status` 欄位（為了維持文件結構的一致性）。
- 不會實作多寶寶切換功能。

## Decisions

### 1. 數據相容性策略 (Ignore on Load)
- **決策**：在 `useEffect` 初始化加載數據以及 `handleImportCSV` 解析數據時，直接將 `type === 'diaper'` 的物件過濾掉。
- **理由**：這樣可以確保 `records` 狀態中永遠只有符合 `RecordType` 定義的合法資料，避免在 UI 渲染時需要處理 `undefined` 或無效類型的情況。

### 2. 統計佈局 (Option B Layout)
- **決策**：使用 Tailwind 的 Grid 佈局將統計區塊分為兩列。
    - 第一列：`col-span-2` 的全寬奶量卡片。
    - 第二列：兩張等寬卡片，分別顯示「睡眠」與「最新成長」。
- **理由**：原本三張小卡片（奶、尿、睡）並排在窄屏（手機）上顯得擁擠且資訊量不足。全寬的奶量卡片可以提供更清晰的視覺層次，也符合「方案 B」的需求。

### 3. 最新成長數據摘要
- **決策**：在 `stats` 的 `useMemo` 中，找出最近一筆 `type === 'growth'` 的紀錄，提取其 `weight` 與 `height`。
- **理由**：讓使用者不必切換到統計頁面或滾動列表，就能直接在首頁看到寶寶最新的發育狀態。

## Risks / Trade-offs

- **[Risk] 舊數據遺失**：使用者過去辛苦紀錄的尿布數據將不再顯示。
    - 🛡️ **Mitigation**: 在提案中明確標示這是 **BREAKING CHANGE**。由於本專案為本地應用，若使用者需要這些數據，建議在更新前先匯出 CSV。
- **[Trade-off] 代碼清理**：雖然 `Record` 介面中的 `status` 屬性目前僅由尿布使用，但考量到未來擴展性，暫不移除此屬性定義，僅移除相關邏輯。
